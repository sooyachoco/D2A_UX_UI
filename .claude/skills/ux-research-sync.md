# UX 리서치 데이터 동기화 워크플로

> **목적**: AI가 상상한 페르소나/여정(🔵 가설)을 **실제 리서치 데이터**로 검증·승격한다.
> 출력물은 `refs/ux-research/` 7종 + `refs/collaboration-tracker.md`. 이 산출물은
> `ai-usability-test`·`ui-design-workflow`·`design:design-critique`가 **단일 source**로 읽는다.
> 신뢰도 표기 규약은 `refs/ux-research/README.md`를 따른다.

---

## 0. 제1원칙 — 출처보다 "전이 적합성"이 먼저다

실제 데이터를 찾는 것보다, **그 데이터가 이 서비스에 유의미한지** 판정하는 게 핵심이다.
도메인이 다른 데이터를 무비판적으로 🟢로 달면 "검증된 것처럼 보이는 가설"이 되어 더 위험하다.

**데이터 우선순위 (높을수록 강함):**

| 등급 | 정의 | 신뢰도 태그 |
|---|---|---|
| 1차 | **대상 서비스 자체**의 사용자 데이터(설문·UT·트래킹) | 🟢 검증 |
| 인접 | 같은 회사 인접 도메인 데이터를 **행동 패턴 수준**으로 전이 | 🟢 인접 실증 |
| 외부 | 3자 플랫폼 공개 데이터·상용 API | 🔵 (벤치마크 전용, 검증 근거 아님) |

> **오버클레임 가드**: 인접 데이터는 "원리"만 전이하고 "수치·시간 척도·도메인 특이 메커닉"은 버린다.
> 게임 진행 목표·파밍·가챠 같은 도메인 고유 개념을 스트리밍 등 다른 도메인에 옮기지 않는다.
> 판정이 애매하면 🔵로 둔다. 한번 단 🟢도 근거가 약하면 **강등하고 트래커에 기록**한다.

---

## Step 0 — 게이트: 대상 서비스 + MCP 소스 확인

1. **대상 서비스 확정**: `PERSONA.md`의 "대상 서비스" 또는 `spec.md`에서 확인. 불명확하면 사용자에게 질문.
2. **MCP 소스 가용성 확인**: 어떤 리서치 데이터가 어느 MCP에 있는지 추측하지 말고 사용자에게 묻는다
   (Notion / SharePoint·Outlook / Slack / GitLab 등). `AskUserQuestion`으로 소스를 좁힌다.
3. MCP 도구가 비활성이면 `mcp-registry`로 후보를 찾아 `suggest_connectors`로 연결 유도.

> 결정 전 반드시 확인(CLAUDE.md ask-before-decide): 데이터 소스·반영 범위·도메인 정렬 방식은 추측 금지.

---

## Step 1 — 소스 탐색 (다단계 검색)

대상 서비스 키워드로 연결된 MCP를 검색한다. **1차 → 인접 → 외부 순**으로 탐색한다.

1. **1차 데이터 탐색** (최우선): "{서비스명} 사용자 설문 / UT / 사용성 / 시청자 리서치" 등.
   서비스명 트리(예: Notion ancestor-path)에 대상 서비스가 있으면 1차일 확률 높음.
2. **인접 데이터 탐색**: 같은 회사의 인접 도메인 리서치(페르소나 프레임워크·장기 테스트 등).
3. **외부 탐색** (보조): `WebSearch` + `mcp-registry`로 공개 데이터셋·API 확인.
   → 즉시 연결 MCP가 없으면 "벤치마크 전용"으로만 다룬다 (Step 6).

검색은 한 쿼리에 한 주제. 소스가 안 나오면 키워드를 바꿔 재시도하고, 없으면 사용자에게 보고.

---

## Step 2 — 소스 검증 (열어서 분류)

찾은 페이지/문서를 **반드시 fetch해서 내용을 확인**한다. 제목·하이라이트만으로 판단 금지.

각 소스를 3종으로 분류한다:

| 분류 | 의미 | 활용 |
|---|---|---|
| **Findings** | 실제 응답·발견·인사이트 (유저보이스, 만족도, 페인포인트) | 🟢 근거로 사용 |
| **스키마/지표 정의** | 받을/측정할 데이터 항목 목록 (아직 결과 아님) | 정량 지표 **정의**로만 사용 |
| **제안/요청 리스트** | 사업 제안·데이터 협업 요청 (결과 아님) | 근거 아님 — 맥락 참고만 |

**표본 한계 파악 필수**: 응답자 수(n)·구성(내부 임직원 vs 외부 유저)·환경(PC/모바일)·기간.
한계는 그대로 기록한다 (예: "사내 도그푸딩·PC 90%+ → 외부·모바일 미검증").

---

## Step 3 — 신뢰도 분류 + 전이 적합성 판정

Step 2의 Findings를 페르소나/여정 항목에 매핑하며, 각 항목에 신뢰도를 부여한다:

- 대상 서비스 1차 Findings로 직접 확인 → **🟢 검증** (+ 표본 한계 주석)
- 인접 도메인 행동 패턴 전이 (Step 0 가드 통과) → **🟢 인접 실증** (+ 전이 가정 주석)
- 근거 없음/전이 부적합 → **🔵 가설**

**전이 적합성 체크리스트 (인접 데이터에만 적용):**
- [ ] 행동 패턴 수준인가? (도메인 고유 메커닉이 아닌가)
- [ ] 시간 척도·수치를 차용하지 않는가?
- [ ] 대상 서비스 맥락에서 말이 되는가?
하나라도 미달이면 🔵로 강등.

---

## Step 4 — 7종 주입 (실증 링크 + 한계 명시)

분류 결과를 `refs/ux-research/` 7종에 반영한다. **단일 source 규약**(README) 준수 — 페르소나/여정은
PERSONA.md / USER_JOURNEY_MAP.md 에서만 정의하고 하위 스킬은 읽기만 한다.

| 파일 | 주입 내용 |
|---|---|
| `PERSONA.md` | P1/P2/P3 항목별 신뢰도 + 실증 링크 + 표본 한계. 강등 시 사유 명시 |
| `USER_JOURNEY_MAP.md` | 단계별 이탈 위험에 신뢰도 컬럼 + 근거 인용 |
| `USER_RESEARCH.md` | "실증 데이터 출처" 섹션(S1·S2…) + 가설 H{n} 상태 갱신 + 마일스톤 |
| `INTERVIEW_GUIDE.md` | 수집 인프라(넥슨퍼스트 패널 등) + Findings를 인터뷰 주제 시드로 |
| `INTERVIEW_NOTES.md` | 1차/인접 참조(REF-*) — 유저보이스 원문 + 출처 링크 보존 |
| `SURVEY_ANALYSIS.md` | 실제 정량 지표(만족도 분포 등) + 트래킹 스키마 + 주관식 패턴 테마 |
| `EMPATHY_MAP.md` | 페르소나별 차원에 신뢰도 + 1차 유저보이스 |

각 실증 항목에는 **출처 링크**(Notion/Confluence URL)와 **인용 원문**을 단다.

## Step 5 — collaboration-tracker 갱신

`refs/collaboration-tracker.md`에 기록한다 (CLAUDE.md update-collaboration-tracker 규칙):

- **실증 데이터 출처** 표 (S{n}: 소스·등급·비고)
- **전환 이력**: 🔵→🟢 승격, 🟢→🔵 **강등**(오버클레임 보정), 신규 가설
- **🟡 확인 필요**: 외부 일반화·모바일 보강·퍼널 트래킹·접근성 등 후속 작업 + 담당/경로

## Step 6 — 외부 소스 분리 (벤치마크 전용)

외부 3자 데이터가 있으면 `refs/ux-research/EXTERNAL_BENCHMARKS.md`에 **분리** 보관한다.
- 🔵 벤치마크로만 라벨링 — 페르소나 검증 근거로 쓰지 않는다.
- 라이브 MCP 연동(API 키·비용 발생)은 사용자 승인 전까지 **미착수**로 둔다 (카테고리 A 결정).

## Step 7 — UX/UI 단계 핸드오프

7종이 갱신되면 다운스트림 스킬이 자동으로 실증 근거를 읽는다:

| 다음 스킬 | 소비 방식 |
|---|---|
| `ui-design-workflow` | PERSONA/JOURNEY의 🟢 검증 페인포인트를 화면 설계 우선순위·인터랙션 게이트에 반영 |
| `ai-usability-test` | PERSONA P1/P2/P3 → UT 페르소나 3종, USER_JOURNEY_MAP → UT_SCENARIOS |
| `design:design-critique` | 페르소나/여정 근거로 휴리스틱 평가 |

> 🔵 가설 항목만으로 UT/설계를 진행할 때는 "가정 기반"임을 산출물에 명시한다.
