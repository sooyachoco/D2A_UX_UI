# UI/UX 설계 워크플로

> **이 문서의 출발점 (왜 필요한가):**
> 선행 작업(0610) 세션에서 시나리오를 아주 구체적으로 잘 썼는데도 메인 기능 화면이 빈 껍데기로 나왔다.
> 원인은 `PRD → 시나리오(산문) → 곧장 코딩`으로 가며 **그 사이의 "화면을 어떻게 설계할지" 단계를 통째로 건너뛴 것**.
> 이 문서는 그 빠진 단계다. 아래 0번 원칙이 전부의 뿌리다.

---

## 0. 제1원칙 — 잘 쓴 시나리오 ≠ 잘 설계된 화면

시나리오는 **커버리지**(모든 단계가 빠짐없이 있다 = "무엇을·왜")를 보장할 뿐,
**인터랙션 충실도**(각 단계가 화면에서 어떻게 보이고 조작되는가 = "어떻게")를 보장하지 않는다.

> "제작자가 메인을 올리면 AI가 비율을 생성한다"는 문장은 *그 단계가 존재한다*만 말한다.
> 그 화면이 어떻게 생겼는지, 4종 소스→44 변형이 한눈에 어떻게 보이는지, 확장 배경이 어떤 컨트롤인지는
> 한 글자도 없다. → 그 자리를 "빈 슬롯 + 버튼"으로 그려도 시나리오를 위반한 게 아니다.

**시나리오가 핵심 인터랙션을 안 정해주면, 구현자는 자기가 아는 가장 일반적인 패턴으로 메꾼다**
(캔버스 = "그냥 Figma 같은 레이어 에디터"). 그 일반 패턴이 이 제품만의 차별점을 지운다.

### 선행 작업(0610) 실패 6원인 (이 워크플로의 각 단계가 막으려는 것)

| # | 실패 | 막는 단계 |
|---|---|---|
| ① | 가장 어려운 부분(AI 생성·확장배경·44변형)을 가장 추상적으로 처리 | §2 깊이 우선 |
| ② | 스펙이 핵심 인터랙션을 안 정해줘 일반 패턴(레이어 에디터)으로 메움 | §3 인터랙션 게이트 |
| ③ | 화면 설계·승인 단계 없이 시나리오→코드 직행, 미시결정 수백 개를 혼자 굳힘 | §4 발산→승인 게이트 |
| ④ | 가장 좋은 레퍼런스(Figma)를 산문으로 요약하며 "어떻게"를 날림 | §5 레퍼런스 키스매드 |
| ⑤ | 9개 화면을 한 번에 만들어 핵심에 1/9 관심만 | §2 깊이 우선 |
| ⑥ | **AI가 기능 흐름을 임의 가정**("메인 1장→4기본형→44변형") → 사용자 소유 위반, 코딩으로 굳힘 | §1.5 0단계 흐름 확정 |

## 1. 핵심 모델 — 순차가 아니라 반복 (dovetail)

산출물은 넷: **목적·요구사항 / 시나리오 / 기능정의 / UI 설계**.
한 번씩 만들고 끝나는 워터폴이 **아니다**. 시나리오와 UI 설계를 거칠 때마다 요구·기능이 함께 더블롭된다.

```
목적+1차 요구  →  시나리오(write-scenario)  →  UI 설계(이 문서)  →  기능정의 확정
                      ↑________ 요구·기능 더블롭 _________↑
```

요구·기능은 시나리오·UI 설계 단계에서 **두 번 갱신**된다. UI 설계 자체가 기능정의를 검증·보강하는 단계다.

## 1.5 단계 모델 — 흐름부터, 싼 것부터 (각 단계 승인 게이트)

UI 작업을 한 번에 코딩으로 밀지 않는다. 아래로 갈수록 비싸지므로(0~2=글·거의 공짜, 5=코드·비쌈)
**일찍·싸게 틀린다.** 각 단계는 사용자 승인 전 다음으로 넘어가지 않는다.

| 단계 | 산출물 | 누가 정하나 | 비용 |
|---|---|---|---|
| **0 · 기능 흐름** | "사용자가 무슨 순서로 무슨 일을 하나" (화면 없음) | **사용자 확정 — AI 추측 금지** | 글 |
| 1 · 화면 도출 | 흐름→화면 목록·연결·통합 | 사용자 승인 | 글 |
| 2 · 화면별 위계 | 1순위/2순위/숨김 + 주 행동 | 합의 | 글 |
| 3 · 레이아웃 발산 | 핵심 화면부터 안 3개(관습 깬 안 1개) | 사용자 선택 | HTML→PNG 목업 |
| **3.5 · 확정 락** | git 태그(`mockup-stage`) + PNG 회귀 baseline + DS 실측 스왑 | 자동(스크립트) | 락 |
| 4 · 상태 설계 | 빈/로딩/완료/수정/실패 | 합의 | 목업 |
| **4.5 · 자가점검 게이트** | 밀도·중복·기준값 점검 → 통과 시 사용자 제출 | AI 자가점검 + 사용자 합의 | 글 |
| 5 · 구현 | 목업→승인→코드 (0-드리프트 전사) | — | 코드 |
| **5.5 · AI UT 게이트** | Playwright + 3 페르소나 + Nielsen 결함 자동 탐지 | 자동 → S4/S3 있으면 사용자 보고 | UT 5종 |

> **0단계가 가장 중요하고 가장 자주 빠진다.** 선행 작업에서 AI가 기능 흐름을 두 번 임의 가정("메인 1장→4기본형→44변형")했고
> 둘 다 틀렸다. **기능 흐름은 사용자 소유다** — AI가 추측해 코딩으로 굳히면 전부 헛수고. 글로 사용자가 확정/교정할 때까지 1단계로 안 넘어간다.
> (이 단계는 dogfooding으로 발견된 구멍 — 원래 이 스킬은 §4 발산으로 바로 갔었다.)

## 1.6 실행 런북 (사전 워크플로우) — 각 스텝: [내 액션] · [사용자 유도] · [게이트]

> §1.5 단계 모델의 실행판. 새 화면/기능 UI를 시작하면 이 순서로 몰고, 각 스텝 끝에 게이트를 받는다.
> **핵심: 사용자가 문제를 *찾게* 하지 말고, 내가 산출물·선택지를 *유도*한다.** (하나하나 잡기 금지)

### STEP 0 · 준비물 확정 (그리기 전 — 80%가 여기서 갈림)
- **내 액션**: 6개 점검 후 *없는 것만* 식별 — ① 비주얼 북극성(톤 레퍼런스 1개) ② 디자인 토큰 단일 출처(`design-direction.md`) ③ PRD 기능=동사+대상 7칸(§5) ④ 메인 기능 1개 ⑤ 상태 목록(빈/로딩/완료/수정/실패) ⑥ 단계 게이트 합의
- **유도**: 없는 것만 콕 집어 청한다 — "톤 레퍼런스 1개 주세요(없으면 제가 극단 3개 던짐) / 메인 기능은 어느 화면?"
- **게이트**: 톤 1개 + 토큰 출처 + 메인 기능 확보 → STEP 1

### STEP 1 · 기능 흐름 확정 (글, 화면 없음)
- **내 액션**: 내가 가정한 흐름을 strawman 다이어그램으로 펼친다
- **유도**: "이 흐름 맞나요? 틀린 화살표·순서를 고쳐주세요" — **AI 추측 금지, 사용자 소유**
- **게이트**: 사용자 확정 → `specs/flow.md` 저장

### STEP 2 · 화면 도출 + 위계 (글)
- **내 액션**: 흐름→화면 목록·연결·통합안 + 각 화면 1순위/2순위/숨김·주 행동
- **유도**: 구조가 갈리는 1~2개만 A/B로 질문(나머진 추천대로 진행)
- **게이트**: 화면 지도 승인

### STEP 3 · 메인 기능 발산 (깊이 우선, 목업)
- **내 액션**: 핵심 화면 레이아웃 안 3개(1개는 관습 깬 안) HTML→PNG 목업(§8)
- **유도**: "어느 안? 섞을 부분?"
- **게이트**: 안 선택 → 더블롭

### STEP 4 · 상태 설계 (목업)
- **내 액션**: 선택 안의 상태 매트릭스(빈/로딩/완료/수정/실패) 목업(§3)
- **유도**: 상태별 화면 확인 + **자가비평**(자신 없는 결정 N개에 깃발 → A/B)
- **게이트**: 합의

### STEP 3.5 · 확정 락 (방향 굳히기)
- **내 액션**: STEP 3 안 선택 직후 (a) `git tag mockup-stage-{NNN}` (b) `design/baseline/{화면}.png` PNG 회귀 baseline 저장 (c) `design/design-direction.md` 의 DS 토큰 값을 §6.5 절차로 **실측 확정**
- **유도**: "방향을 락 합니다. 이후 디테일 핑퐁은 lock 후에만 시작합니다"
- **게이트**: 태그 + baseline PNG 존재 → STEP 4

### STEP 4 · 상태 설계 (목업)
- **내 액션**: 선택 안의 상태 매트릭스(빈/로딩/완료/수정/실패) 목업(§3)
- **유도**: 상태별 화면 확인 + **자가비평**(자신 없는 결정 N개에 깃발 → A/B)
- **게이트**: 합의

### STEP 4.5 · 자가점검 게이트 (시안 제출 전)
- **내 액션**: §6 출력 전 점검 5항(안내 문구·필수 요소·기준 토큰·폭 일치·클래스 충돌) + **밀도/중복** 추가 점검을 **자가비평으로 명시 출력**한다. 통과한 것만 사용자에게 제출.
- **유도**: "다음 항목을 자가점검했습니다 — ✅/⚠️ 표기. ⚠️면 수정 후 다시 제출"
- **게이트**: 자가점검 5/5 ✅ + 밀도/중복 ✅ → 사용자 제출

### STEP 5 · 구현
- **내 액션**: 목업→코드. 토큰만 사용·흐림 방지(§6)·출력 전 점검. **확정 락 baseline PNG와 시각 회귀 1회 비교** (visual regression).
- **게이트**: 빌드 통과 + baseline PNG diff ≤ 임계값

### STEP 5.5 · AI UT 게이트 (배포 전)
- **내 액션**: `ai-usability-test` 스킬 자동 실행 — Playwright + 3 페르소나(라이트/코어/접근성) + Nielsen 휴리스틱
- **유도**: 결함 S4 발견 시 ⛔ 배포 차단 / S3 발견 시 다음 스프린트 후보 보고
- **게이트**: S4 = 0 → 배포 가능. 산출물은 `specs/{NNN}/ut/` 5종.

### ⟲ 사이드레일 — "마음에 안 들 때" (하나하나 잡지 않는다)
1. **한 축으로 좁힌다**: 톤·비주얼 / 위계 / 밀도·여백 / 흐름 / (모르겠으면 자가비평 요청)
2. **축별 프롬프트(복붙)**:

| 축 | 프롬프트 |
|---|---|
| 톤·비주얼 | "전체 톤 안 맞아. `{레퍼런스}` 느낌으로 **북극성 다시, 한 화면만** 그 톤으로 (디테일 말고 톤만)" |
| 위계 | "`{핵심}`을 1순위로 키우고 보조는 눌러서 다시" |
| 밀도·여백 | "여백·카드 간격 **기준 한 번 정하고 전 화면 일괄**" |
| 흐름 | "`flow.md` 기준으로 화면 순서/통합 다시" |
| 모르겠음 | "자신 없는 결정 3개에 **깃발 꽂고 A/B로** 줘" |

3. **2-strike 후퇴**: 같은 화면을 2번 고쳐도 별로면 → 디테일 멈추고 **STEP 0(북극성)으로** 후퇴.
4. **피해야 할 프롬프트**: "전체적으로 톤 정리하고 다 만들어줘" / "뭔가 별로인데 예쁘게" — 축·기준·범위가 없어 핑퐁을 부른다.

## 2. 깊이 우선 — 메인 기능 화면을 가장 먼저, 가장 깊게

**breadth-first 금지.** 9개 화면을 동시에 만들지 않는다. 순서는:

1. **메인 기능 화면 1개**를 먼저 식별한다 = *이 제품의 차별점이자 가장 어려운 화면*.
   (선행 작업 = 에셋 등록의 AI 생성/확장배경, 또는 베리보드의 44변형 — 로그인·대시보드가 아니다.)
2. 그 화면 하나를 §3~§4로 **끝까지** 설계·승인·구현한다.
3. 거기서 굳은 컴포넌트·토큰·패턴을 나머지 화면에 펼친다 (§6 흐림 방지).

> 노력을 거꾸로 쓰지 않는다. 메인 기능에 80%. 로그인·대시보드는 마지막에 관습대로 빠르게.

## 3. 인터랙션 충실도 게이트 — "어떻게"를 상태로 못박는다

메인 기능 화면은 레이아웃만으론 부족하다. **핵심 메커니즘을 상태별로** 설계하고 승인받는다.
빈 슬롯이 나오는 이유는 이 게이트가 없어서다.

### 상태 매트릭스 (코딩 전 반드시 채운다)

핵심 인터랙션마다 아래 상태에서 *화면에 무엇이 보이고, 사용자가 무엇을 누르는지*를 적는다:

| 상태 | 채울 것 |
|---|---|
| **빈(empty)** | 아직 아무것도 없을 때 — 안내 아이콘+CTA 1개 (안내 문구 금지) |
| **로딩·생성중** | AI가 도는 중 — 진행 표시, 무엇이 몇 개 중 몇 개 |
| **결과(채워짐)** | 실제 콘텐츠가 보임 — *플레이스홀더 회색박스가 아니라 진짜처럼* |
| **선택·포커스** | 사용자가 하나 고른 상태 — 무엇이 강조되나 |
| **수정·편집** | 그 결과를 바꾸는 컨트롤 — 어떤 조작(드래그/입력/토글)인가 |
| **완료·승인** | 명시 승인 후 — 배지·상태 전이 |
| **실패·부분** | 일부 실패/충돌 — 어떻게 표시·복구하나 |

> 도구의 본질(이미지·시안)을 다루는 화면이면 **결과 상태를 진짜 콘텐츠로** 그린다.
> CSS 도형/그라데이션으로 "가짜 시안"이라도 만들어 빈 회색박스를 없앤다.

### 정보 위계 (모든 화면 공통)

모든 요소를 같은 크기·굵기·간격으로 나열하지 않는다. **주(1순위) / 보조(2순위) / 참조(숨김·접힘)** 를
크기·굵기·색·여백·그룹화·위치로 구분한다. "무엇을 먼저 보고 무엇을 할지"가 한눈에.

## 3.5 확정 락 — 방향이 정해진 시점에 "굳혀라"

STEP 3에서 안이 선택된 순간, **이후 디테일 핑퐁이 방향을 흐트러뜨리지 못하게** 락을 건다.
이 단계가 빠지면 "이미 정한 방향"이 핑퐁 라운드마다 미세하게 어긋나 드리프트가 누적된다.

### 락 3종 (자동화)

```bash
# (a) git 태그 — 시점 영구 기록

# (c) DS 토큰 실측 (§6.5)
# refs/design-systems/{ds}.md 와 design/design-direction.md 가 일치하는지 검증
```

- **레이아웃·정보 위계·핵심 인터랙션 변경 금지** — 변경하려면 STEP 3로 되돌아가 재발산

- 시각 회귀 임계치를 넘는 변경(예: baseline PNG와 20% 이상 diff)은 자동 차단

## 4. 발산 → 선택 → 더블롭 게이트 (혼자 굳히지 않는다)

미시결정을 혼자 내려 코딩으로 굳히는 게 ③번 실패다. 그래서:

1. **레이아웃 안 3개를 발산.** 그중 **1개는 반드시 관습을 깬 안**(국소최적·국소최적 방지).
   각 안마다 §3의 핵심 인터랙션이 어떻게 보이는지 ASCII 목업 또는 저비용 와이어프레임으로.
2. **사용자가 선택** → 그 안을 더블롭(상세화).
3. **승인 후에만 코딩.** 와이어프레임/목업 → 승인 → 구현. (이 환경에선 React 직행 대신,
   먼저 목업이나 단일 화면 프로토타입을 보여주고 승인받은 뒤 전체 구현.)

> 게이트 통과 신호: 화면의 핵심 인터랙션이 상태별로 정의됐고, 사용자가 레이아웃 안을 골랐을 때.

## 4.5 자가점검 게이트 — 시안 제출 전 5+2항 셀프 체크

사용자에게 시안을 보여주기 *전*에 AI 스스로 점검한다.
이 게이트가 없으면 사람이 매번 같은 종류의 마찰(밀도·중복·기준값 불일치)을 잡아야 한다.

### 자가점검 7항 (시안 제출 전 명시 출력 — 통과한 것만 제출)

| # | 항목 | 통과 기준 |
|---|---|---|
| 1 | 안내 문구 없음 | UI 내부 사용법·설명 텍스트 0건 (아이콘+라벨로) |
| 2 | 필수 요소만 | 장식·중복 0건. 화면당 같은 단어 5회 이상 반복 없음 |
| 3 | 기준 토큰만 | hex 직접 사용 0건. 모든 색·여백·radius가 토큰 변수 |
| 4 | 폭 일치 | 사이드/패널/콘텐츠 폭이 design-direction.md 정의값과 일치 |
| 5 | 클래스/이름 충돌 | 컴포넌트 이름·CSS 클래스 중복 0건 |
| 6 | 밀도 균일 | 카드 간격·섹션 패딩이 화면 내 일관 (편차 ≤ 4px) |
| 7 | 컴포넌트 SSOT | 2회 이상 등장한 UI는 ui-components.md에 등록 |

### 출력 형식 (사용자에게 시안 제출 시 함께)

```
## 자가점검 (시안 제출 전)
✅ 1. 안내 문구 없음
✅ 2. 필수 요소만
⚠️ 3. 기준 토큰만 — Hero CTA hover 색상이 토큰 외 (#3056d6 직접 입력) → 수정 후 재제출
✅ 4. 폭 일치
✅ 5. 클래스/이름 충돌 없음
✅ 6. 밀도 균일
⚠️ 7. 컴포넌트 SSOT — "라이브 카드"가 2회 등장 → ui-components.md 승격 필요
```

⚠️가 1개라도 있으면 시안 제출 전 자동 수정·재점검. 5/7 미만이면 STEP 4로 되돌아간다.

## 5. 레퍼런스 키스매드 — "어떻게"의 최고 출처를 산문으로 날리지 않는다

레퍼런스(Figma·기존 서비스·와이어프레임)가 있으면 **그 화면 자체를 설계 기준으로 삼는다.**
④번 실패 = Figma의 실제 메커니즘을 "비교"하고 "산문 시나리오에 파편 머지"만 해서 어떻게가 뭉개진 것.

- 레퍼런스는 **베끼지 않는다.** 그 키스매드(레이아웃·정보 위계·여백·텍스트 크기차·진입점)를 가져와 맥락에 맞춘다.
- **레퍼런스의 요소를 그대로 베끼지 않는다** — "그 요소가 우리 맥락에서도 의미가 있나"를 먼저 거른다.
  (예: 증권 시세 화면의 종목 태그는 분류용이었지만, 전부 같은 종류인 화면에 그대로 넣으면 무의미한 군더더기.)
- Figma가 있으면 MCP(`get_screenshot`·`get_design_context`)로 **직접 보고** 그 화면을 스펙으로 쓴다.
  산문으로 요약해서 넘기지 않는다.
- **레퍼런스는 PRD의 기능(job)별로 쪼개 요청한다.** 복합 툴은 한 제품이 전부를 커버하지 않으므로
  "○○ 툴" 한 덩어리로 찾지 말고, ① 전체 비주얼 톤 1개(북극성) + ② 기능별 인터랙션 레퍼런스(화면별)로 나눈다.
  사용자에게 레퍼런스를 청할 때 PRD에서 job을 뽑아 "이런 종류를 보여달라"고 **카테고리로 제안**하고,
  **각 카테고리마다 적합한 구체 제품을 직접 추천**한다 — "레퍼런스 주세요"라고 백지로 묻지 말 것.
  사용자가 떠올리게 하지 말고, 내가 PRD 기반으로 골라 제시하면 사용자는 *고르거나 반례를 들기*만 하면 된다(부담↓·수렴↑).

### 레퍼런스가 잘 잡히는 PRD 입력란 (채우면 톤·기능별 레퍼런스가 자동 매칭)

레퍼런스 품질은 **PRD가 기능을 "인식 가능한 job"으로 적었는가**에 비례한다. PRD(또는 `/refine-prd` 단계)에서 아래 7칸을 채운다.
**핵심: 기능을 "동사 + 대상"으로 적는다.** 명사·모호어("관리/처리")로 적으면 매칭 불가.

```
1. 제품 유형·맥락 : [사내/외부] · [B2B/B2C] · [웹/모바일] · [사용 환경]      → 톤 북극성
2. 주 사용자      : [역할] · [사용 빈도] · [숙련도] · [핵심 목표]            → 밀도·톤
3. 디자인 제약    : [디자인 시스템] · [주요 색·폰트] · [지켜야 할 룩]          → 톤 후보 정렬
4. 핵심 기능      : 각 줄 "[동사] [대상] — [핵심 인터랙션 1줄]"  ★메인 기능 표시 → 기능별 레퍼런스 + 깊이
5. 데이터 규모    : [단건/다수] · [얼마나 — 예: 매체×사이즈 44변형]          → 그리드/대시보드 필요 여부
6. 유사 제품(선택): "○○처럼 ~"                                              → 있으면 역산 없이 직격
```

**동사+대상 리라이트 (이게 핵심 — 같은 기능도 이렇게 바꿔 적어야 매칭됨):**

| ❌ 명사·모호 | ✅ 동사 + 대상 + 인터랙션 |
|---|---|
| 에셋을 관리한다 | 항목별 메인 에셋을 등록하고 AI로 비율별 이미지를 생성·프롬프트로 수정한다 |
| 검수 기능 | 검수자가 배너 단위로 승인/반려하고 코멘트를 단다 |
| 배너 변형 | 1장을 매체·사이즈별 44개로 자동 배치한다 |

## 5.5 비주얼 북극성과 핑퐁 순서 — 방향 확정 전 디테일 수정 금지

규칙:
- **비주얼 북극성을 디테일보다 먼저.** "마음에 드는" 레퍼런스 1개를 확보하거나(기존 서비스·무드보드),
  *극단적으로 다른* 비주얼 방향 2~3개를 빠르게 던져 고르게 한다. 고른 것이 타깃.
- **핑퐁(디테일 1:1 수정)은 방향이 확정된 뒤 마지막에만.** 핑퐁은 *방향이 맞다는 전제*에서 다듬는 도구다.
  레이아웃 컨셉·비주얼이 안 맞으면 핑퐁은 영원히 수렴하지 않는다.
- **"별로"를 구조화한다.** 막연한 "마음에 안 듦"을 한 축(가독성/위계/톤/밀도)으로 짚게 해 원인을 좁힌다.
- **2-strike 후퇴.** "또 마음에 안 듦"이 2회 연속이면 디테일을 멈추고 방향·북극성 단계로 후퇴한다.

## 6. 흐림(drift) 방지 체계 — 화면이 늘어도 톤이 안 깨진다

화면마다 같은 컴포넌트를 다시 그리면 탭·여백·토큰이 어긋난다(=곳곳 미세 차이 = "전반적으로 안 듦").

1. **공통 기준 파일 하나.** 디자인 토큰(색·폰트·간격) + 레이아웃 폭(사이드/패널/콘텐츠) + 공통 컴포넌트를
   단일 출처(CSS 변수 또는 토큰 파일)로. 모든 화면이 그걸 불러쓴다.
2. **2회 이상 반복되는 UI는 즉시 컴포넌트로 승격.** 처음부터 전체 컴포넌트 시스템을 설계하지 않는다(무거움) —
   *두 번째 등장하는 순간* 공통 조각으로 뺀다.
3. **같은 화면의 상태 variant는 별 파일 복붙이 아니라 한 함수에서 플래그로 분기.** (복붙하면 한쪽만 고쳐져 어긋난다.)
4. **레이아웃 섹션도 토큰이다** — 사이드(LNB) 폭, 우측 패널 폭, 본문 패딩, 콘텐츠 최대폭. 화면마다 px로 주면
   곳곳이 미세하게 어긋난다. 첫 화면에서 한 번 정해 토큰으로.
5. **출력 전 점검** ①안내 문구 없음(아이콘+라벨) ②필수 요소만(장식·중복 제거) ③기준 토큰만 ④폭 일치 ⑤클래스/이름 충돌 검사.
6. **시각 디테일(여백)은 토큰으로 일관.** 단 '정렬 버그'(중앙정렬이 안 먹는 등)는 즉시 고친다 — 여백 기준값과 다른 문제.

### 추가 규칙 (이전 테스트 인사이트)

- **안내 문구 금지** — UI 내부 사용법·설명 텍스트 안 넣는다. 아이콘+라벨로. (단 빈 상태 안내는 화면별 판단)
- **메뉴/네비는 '이름'이 아니라 '사용자가 누르면 뭘 기대하는가'로 정의** — 겹침·모호함이 드러난다.
- **최상위 네비(GNB)엔 '영역'만, 분류는 LNB가 맡는다** — 메뉴 늘면 GNB에 추가하지 말고 LNB 분류로 흡수.
- **요청한 부분만 고친다** — 요청 안 한 색·레이아웃·여백·크기 안 건드린다. 전체 다시 짜다 다른 곳 깨뜨리지 말 것.
- **보기 모드 탭은 같은 데이터 집합을 보는 방식만 바꾼다** — 탭마다 항목 개수가 달라지면 안 된다.
- **한 화면에 같은 단어 5~7번 반복은 군더더기** — 제목·CTA 등 꼭 필요한 곳만.
- **중복 화면 경고** — 어떤 단계 화면이 다음 화면과 거의 겹치면 별 화면 대신 버튼+상태로 통합.
- **미결·충돌을 가시화** — 와이어프레임이 정교해지면 미정의(좌석/연동)·충돌이 "다 된 것처럼" 묻힌다. 진행현황에 '미정의/충돌' 상태로 남긴다.

## 6.5 DS 실측 — 가정 색상 ≠ 실제 색상 트랩 차단

토큰을 글로만 채우면 가정과 실제가 어긋난다 (선행 작업에서 가정 `#355bf7` ≠ 실제 `#57a0ff` 발견).
**스왑 패스(swap pass)** 로 실측한 뒤에만 락을 건다.

### 스왑 패스 절차

```bash
# 1. design-direction.md 에 추측된 토큰 값을 적는다 (예: color-pc-500 = #355bf7)
# 2. 실제 DS 출처에서 같은 토큰을 자동 추출
#    - NX Basic 1.0v: nxbasic-mcp `search_design_tokens` 우선 (정확한 토큰명·값 반환)
#      예) search_design_tokens(query="pc", category="colors") → color-pc-500=#6babff
#      폴백: refs/design-systems/nxbasic-1.0v.md 또는 Storybook WebFetch
#    - 외부 DS: design/samples.html 의 실제 CSS 변수 인스펙트
# 3. 두 값을 비교 → 다르면 design-direction.md 값을 실제값으로 교체
#    ※ 토큰명을 step 번호까지 정확히 (pc-500 ≠ pc-600). MCP가 이 라벨 오류를 예방한다.
# 4. 교체 이력을 design/baseline/ds-swap-log.md 에 기록
```

### 스왑 로그 형식 (`design/baseline/ds-swap-log.md`)

```markdown
| 토큰 | 가정값 | 실측값 | 출처 | 날짜 |
|---|---|---|---|---|
| color-pc-500 | #355bf7 | #6babff | nxbasic-mcp | 2026-06-24 |
| color-pc-600 | #57a0ff | (일치) | nxbasic-mcp | 2026-06-24 |
```

## 9. 내게 일을 잘 시키는 프롬프트 (사용자 질문 답)

핵심: **WHAT(무엇을)만 주면 나는 일반 패턴으로 메꾼다. HOW-제약(상태·레퍼런스·위계·게이트)을 줘야 잘한다.**

### ✅ 잘 되는 프롬프트 — 3가지 패턴

**(A) 발산 요청** — 설계 초기, 방향이 안 정해졌을 때:
```
{화면} 레이아웃 안을 3개 내봐. 1개는 관습을 깬 안으로.
각 안마다 핵심 인터랙션({메커니즘, 예: AI 비율 생성→확장배경})이
어떻게 보이는지 ASCII 목업으로. 코딩하지 말고 안만 보여줘.
```

**(B) 인터랙션 못박기** — 방향은 정해졌고 핵심 화면을 깊게:
```
{화면}의 {핵심 기능}을 상태별로 설계해줘:
빈 / 생성중 / 생성완료 / 수정중 / 완료 / 실패.
각 상태에서 화면에 뭐가 보이고 사용자가 뭘 누르는지.
레퍼런스는 {Figma node 또는 화면}. 목업 먼저 보여주고, 승인하면 코딩.
```

**(C) 단일 화면 깊게 + 범위 보호**:
```
다른 화면 건드리지 말고 {화면} 하나만.
{레퍼런스} 수준으로. 토큰은 {기준 파일}만 사용.
요청 안 한 부분은 그대로 둬.
```

### 좋은 프롬프트의 6요소 (체크리스트)
1. **화면 1개** 지정 (핵심·최난도 먼저 — breadth 금지)
2. **목적 + 핵심 인터랙션** 명시 (또는 "안 3개 발산해")
3. **기준**: 레퍼런스 화면 또는 토큰 파일
4. **상태들**: 채워야 할 상태 (빈/로딩/선택/완료/실패)
5. **승인 게이트**: "코딩 전 목업 먼저, 승인하면 구현"
6. **범위 보호**: "요청 안 한 부분 건드리지 마"

## 10. 관련 문서·추적

- `write-scenario` 스킬 — 이 문서의 **앞 단계**(시나리오=무엇을·위계). 이 문서는 그 다음(어떻게).
- `scenario-fidelity` 렌즈 — 시나리오 *커버리지* 검증. **인터랙션 품질은 못 잡는다**(그래서 이 문서가 필요).
- 규칙 효과 추적(A~F 지표: 재시작/구조 선작성/대안 다양성/번복/미결 추적/검증 적용) — 규칙이 마찰만 주고
  효과 없으면 폐기, 효과 확인되면 강화. 작업 단위마다 BEFORE→AFTER로 방향성을 본다.
