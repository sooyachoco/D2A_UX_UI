# Check Decision Gates

구현 시작 전에 `specs/{프로젝트}/decisions.md`의 미결정(⬜) 항목을 확인하고,
현재 작업에 필요한 항목이 있으면 사용자에게 질문한다.

## Step 1: decisions.md 읽기

decisions.md를 읽어 전체 상태를 파악한다.

상태 정의:
- ✅ 확정: 구현 가능
- ⬜ 미결정: 구현 차단 — 반드시 질문
- 🔄 재검토: 진행하되 사용자에게 알림
- ❌ 거부: 하지 않기로 한 것

## Step 2: 현재 작업과 대조

사용자가 요청한 작업(Phase, Task)에 필요한 항목을 필터링한다.

```
## Decision Gate 체크 결과

### ✅ 확정된 항목 ({N}개)
- G-001: Auth 모듈 → internal-sso
...

### ⬜ 미결정 항목 ({N}개) — 결정이 필요합니다
- G-012: DB 종류/버전
...
```

## Step 3: 정책 기반 자동 선택 시도

⬜ 항목마다 refs/INDEX.md → refs/policies/ 순서로 자동 선택을 시도한다:
- 정책 근거 있으면: decisions.md ✅ 갱신 + `"🤖 정책 자동 선택 (근거: {파일})"` 표시 → 인라인 보고: `"💡 [{항목}] → [{값}]으로 자동 선택"`
- 정책 근거 없거나 충돌하면: "사용자 판단 필요" 목록에 추가

**사용자 판단 필요 항목만 질문한다:**

```
🔶 결정이 필요합니다: {항목명}

**상황**: {왜 이 결정이 필요한지}
**선택지**:
  A) {선택지 1} — {장단점}
  B) {선택지 2} — {장단점}
  C) {기타 — 직접 지정}

**참고**: {정책 정보 없음 또는 정책에 충돌하는 선택지}
**영향**: {이 결정이 미치는 영향}
```

자동 선택으로 ⬜ 항목이 모두 해결되면 사용자 질문 없이 구현을 시작한다.

## Step 4: 기록 업데이트

사용자 답변을 받으면:
1. decisions.md 해당 항목을 ✅로 변경
2. 결정 내용과 결정일을 기록
3. 관련 spec.md가 있으면 동기화

```bash
./scripts/log-activity.sh DECISION "{항목ID}: {결정 내용 요약}" "" || true
```

## Step 5: Prerequisites 교차 검증

decisions.md 체크 후, `specs/{프로젝트}/prerequisites.md`가 존재하면 교차 검증한다.

🔴 미해결 항목이 있으면 해당 시스템 관련 기능 구현을 차단한다.

```
## Prerequisites 교차 검증

| 시스템 | 관련 Phase | 미해결 🔴 | 구현 영향 |
|---|---|---|---|
| {시스템} | Phase {N} | {N}개 | ⛔ 차단 / ✅ 통과 |
```

## Step 6: 게이트 통과 확인

```
## Gate 체크 완료

### Decision Gates
- ✅ 확정: {N}개
- ⬜ 미결정: 0개

### Prerequisites
- ✅ 확인: {N}개
- ⛔ 차단: {N}개

{⬜ = 0 && ⛔ = 0}
모든 필수 Gate를 통과했습니다. 구현을 시작합니다.
```

⬜가 0이 되어야 구현을 시작한다.
⛔ 항목이 있으면 해당 시스템 관련 구현을 차단한다.
사용자가 "나중에 결정할게"라고 하면 해당 부분의 코드는 TODO로 남긴다.
