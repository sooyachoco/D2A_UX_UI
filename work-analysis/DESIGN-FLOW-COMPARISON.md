# DESIGN-FLOW-COMPARISON — 디폴트 vs 실제 작업 흐름 갭 분석

> **출처**: 선행 작업 분석 (2026-06-19) + NEXON STREAM 작업 검증 (2026-06-23)
> **목적**: 디폴트 흐름(template + skills)이 실제 효과적 흐름과 어디가 어긋났는지 1:1 매핑.

## 비교표 (11개 갭)

| # | 항목 | 🔵 기존 디폴트 | 🟢 실제 효과적 흐름 | 얻은 것 |
|---|---|---|---|---|
| 1 | **화면 도출 입력** | Q6\* 디자인 샘플 3종 선택 + 기본 정보 | PRD → 역할 몰입 시나리오 v3(위계 명시) → 화면 | 화면이 "예쁜 틀"이 아니라 *일하는 흐름*에서 나옴 |
| 2 | **목업 매체** | 곧장 React 프로토타입 (더미데이터) | HTML→PNG 목업 먼저, React는 나중 | 빠르고 토큰 효율적, 디자인만 빠르게 반복 |
| 3 | **수렴 방법** | 핑퐁 (암묵·규칙 없음) | 발산 3안 → 근거 선택 → 차점자 이식 | 5라운드 핑퐁 → 0라운드 수렴 |
| 4 | **확장 방법** | 화면마다 개별 | 앵커 1화면 후 시스템 경량 루프 | 일관성 + 속도 동시 |
| 5 | **방향 합의** | 없음 (바로 그림) | 0단계 게이트 (정의→승인→그리기) | 헛수고·재작업 감소 |
| 6 | **컴포넌트화** | design-direction "커스터마이징 방향" 칸만 | 2회 재사용 시 SSOT 승격 (ui-components.md) | 드리프트 감소 |
| 7 | **확정/락** | 암묵 (명시 절차 없음) | git 태그(mockup-stage) + PNG 회귀 baseline | "그대로 구현"을 검증 가능하게 |
| 8 | **React 전환** | 처음부터 React | 확정·락 후 0-드리프트 전사 | 디자인 흔들릴 때 코드 안 함 → 재작업비 절감 |
| 9 | **DS 적용** | 색상 등 값을 추측 입력 | 스왑 패스로 실측 | 가정 `#355bf7` ≠ 실제 `#57a0ff` 발견 |
| 10 | **레퍼런스 도출** | Q6\* 샘플 3종을 먼저 제시 → 사용자가 선택 (레퍼런스→제품) | PRD 입력 → 유저 시나리오 학습 → AI가 맥락 맞는 레퍼런스 역으로 제안 (제품→레퍼런스) | 블라인드 선택 → 제품 맥락 근거, 적중률 ↑ |
| 11 | **레퍼런스 사용 범위** | 샘플을 통째로 차용 | 비주얼만 차용, 조작 모델은 스펙에서 | 제품 고유 메커니즘 안 망가짐 |
